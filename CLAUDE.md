# CLAUDE.md — Fleet Management Platform (MVP Demo)

> This is the MVP demo stack (Vercel + Supabase, zero infrastructure cost).
> When the project graduates to production on AWS, switch to CLAUDE.enterprise.md.

---

## Purpose

Stakeholder demo to validate the product concept before AWS investment.
Target: 50–100 simulated vehicles, a small number of demo users, no real
telematics integration. All vehicle movement is generated by a simulator script.
Demo scenario: 50 vehicles operating across Gothenburg, Sweden — a mix of long-haul
trucks on the E6/E45 corridors and city buses on urban routes. Seed data
should reflect this geography.

**Do not over-engineer this.** Every decision should favour shipping speed over
scalability. The enterprise patterns (microservices, Kinesis, Redis, Timestream)
are documented in CLAUDE.enterprise.md — do not introduce them here.

## Tech Stack

| Concern | Choice | Why |
|---|---|---|
| Framework | Next.js 15 (App Router) | One repo for frontend + API routes |
| Language | TypeScript (strict mode) | Shared types, fewer runtime surprises |
| Database | Supabase (PostgreSQL) | Free tier, instant setup, Realtime built in |
| ORM | Prisma | Type-safe queries, easy migrations |
| Real-time | Supabase Realtime | Broadcasts position updates; no WebSocket server needed |
| Auth | Supabase Auth | Email/password + Google OAuth for demo |

| UI | shadcn/ui + Tailwind CSS v4 | Consistent with production target |
| Charts | Apache ECharts | Dashboards and driver performance charts |
| Data tables | TanStack Table v8 | Virtualised lists for vehicles and drivers |
| Server state | TanStack Query v5 | Consistent with production target |
| Client state | Zustand | Selected vehicles, map viewport, active filters |
| Forms | React Hook Form + Zod | Schemas reusable in production |
| Hosting | Vercel (hobby tier) | Free, zero-config Next.js deploy |
| Simulator | Node.js seed script | Moves vehicles realistically; runs locally or as a Vercel cron |

## Architecture

Everything lives inside one Next.js application. No separate backend services.

```
Browser
  └── Next.js app (Vercel)
        ├── app/             ← pages, layouts, server components
        ├── app/api/         ← API routes (replace microservices)
        └── lib/             ← shared logic, Supabase client, Prisma client

Supabase (hosted PostgreSQL)
  ├── Tables                 ← all domain data
  ├── Realtime               ← broadcasts position updates to the browser
  └── Auth                   ← user sessions, Google OAuth

Simulator script (scripts/simulate.ts)
  └── Runs locally: updates vehicle positions in Supabase every 5s
```

### How real-time works in the MVP

The simulator script updates the `positions` table in Supabase every 5 seconds.
Supabase Realtime broadcasts the change to subscribed browser clients.
The map re-renders on each broadcast. No WebSocket server, no Redis, no Kinesis.

In production this will be replaced by the Kinesis → WebSocket Gateway pipeline,
but the browser-side subscription code stays almost identical.

## Database Schema (Prisma)

Core tables only — no audit log, no separate time-series store.
Position history lives in the `positions` table with a standard index on `created_at`.

```prisma
model Organisation {
  id        String   @id @default(uuid())
  name      String
  createdAt DateTime @default(now())
  fleets    Fleet[]
  users     User[]
}

model Fleet {
  id       String       @id @default(uuid())
  name     String
  orgId    String
  org      Organisation @relation(fields: [orgId], references: [id])
  vehicles Vehicle[]
}

model Vehicle {
  id           String            @id @default(uuid())
  registration String            @unique
  type         String            // "truck" | "bus" | "van"
  make         String
  model        String
  year         Int
  fleetId      String
  fleet        Fleet             @relation(fields: [fleetId], references: [id])
  driverId     String?
  driver       Driver?           @relation(fields: [driverId], references: [id])
  positions    Position[]
  fuelRecords  FuelRecord[]
  services     ServiceRecord[]
  schedules    ServiceSchedule[]
}

model Driver {
  id       String        @id @default(uuid())
  name     String
  licence  String
  phone    String
  email    String
  orgId    String
  vehicles Vehicle[]
  scores   DriverScore[]
}

model Position {
  id        String   @id @default(uuid())
  vehicleId String
  vehicle   Vehicle  @relation(fields: [vehicleId], references: [id])
  lat       Float
  lng       Float
  heading   Float
  speedKmh  Float
  engineOn  Boolean
  createdAt DateTime @default(now())

  @@index([vehicleId, createdAt])
}

model FuelRecord {
  id        String   @id @default(uuid())
  vehicleId String
  vehicle   Vehicle  @relation(fields: [vehicleId], references: [id])
  litres    Float
  costEur   Float
  odometer  Float
  createdAt DateTime @default(now())
}

model ServiceRecord {
  id          String   @id @default(uuid())
  vehicleId   String
  vehicle     Vehicle  @relation(fields: [vehicleId], references: [id])
  description String
  costEur     Float
  workshop    String
  completedAt DateTime
}

model ServiceSchedule {
  id          String   @id @default(uuid())
  vehicleId   String
  vehicle     Vehicle  @relation(fields: [vehicleId], references: [id])
  description String
  dueAt       DateTime
  dueMileage  Float?
  done        Boolean  @default(false)
}

model DriverScore {
  id           String @id @default(uuid())
  driverId     String
  driver       Driver @relation(fields: [driverId], references: [id])
  period       String // "2025-W12" (ISO week) or "2025-04" (month)
  score        Float  // 0–100
  harshBraking Int
  harshAccel   Int
  speeding     Int
  idleMinutes  Int
}
```

## Essential Commands

```bash
# Install dependencies
pnpm install

# Start dev server
pnpm dev

# Run the vehicle simulator (separate terminal)
pnpm simulate

# Push schema changes to Supabase (dev)
pnpm db:push

# Generate Prisma client after schema changes
pnpm db:generate

# Create a new migration
pnpm db:migrate

# Seed demo data
pnpm db:seed

# Type-check
pnpm typecheck

# Lint
pnpm lint

# Build
pnpm build
```

Add to `package.json` scripts:
```json
{
  "simulate":    "tsx scripts/simulate.ts",

  "db:generate": "prisma generate",
  "db:migrate":  "prisma migrate dev",
  "db:seed":     "tsx prisma/seed.ts"
}
```

## Simulator Script

`scripts/simulate.ts` moves vehicles along realistic routes every 5 seconds.
It must be the only writer to the `positions` table during a demo session.

Behaviour:
- Loads all vehicles from the database on startup
- Each vehicle follows a pre-defined waypoint loop (city + highway mix)
- Speed varies by vehicle type and road segment
- `engineOn` toggles realistically (vehicles stop for 2–5 min occasionally)
- Runs until killed with Ctrl+C

A stateless Vercel Cron endpoint exists at `app/api/cron/simulate/route.ts`
for Pro plan deployments. On the Hobby plan the per-minute schedule is blocked —
remove the `crons` block from `vercel.json` and run the simulator locally instead.

## Coding Conventions

### General
- TypeScript strict mode — no `any`, no `@ts-ignore`
- `const` by default; `let` only when reassignment is needed; never `var`
- `async/await` over `.then()` chains
- Zod schemas in `types/index.ts` — single source of truth for API validation

### Next.js patterns
- Server Components by default; add `"use client"` only when hooks or browser
  APIs are needed
- All API routes in `app/api/` — no separate Express/Fastify server
- Use the server Supabase client (`lib/supabase/server.ts`) in Server
  Components and API routes; browser client only in `"use client"` components
- Prefer server-side data fetching for non-realtime data (vehicle list, driver
  profiles, service history)

### API routes
All responses use this shape:
```json
{ "data": { ... }, "error": null }
{ "data": null, "error": "Something went wrong" }
```
Error messages must be user-readable strings — never raw Prisma/PostgreSQL errors.

### Naming
- Files: `kebab-case`
- Components: `PascalCase`
- Functions/variables: `camelCase`
- Database tables/columns: `snake_case` (Prisma maps automatically)

### Imports
- Use `@/` alias for all internal imports (configured in `tsconfig.json`)
- No relative `../../` imports

### Styling
- Tailwind utility classes only — no inline styles, no CSS modules
- No hardcoded colour hex values — use Tailwind design tokens
- Mobile-first: mobile styles default, `md:` and `lg:` for larger screens

## Architectural Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Single Next.js app | Monolith | Fastest to ship; split into services post-validation |
| Supabase Realtime | Instead of WebSocket server | Zero infrastructure; same browser-side concept as production |
| Positions in PostgreSQL | Instead of Timestream | Sufficient for 50–100 vehicles; swap post-MVP |
| OSM tiles | Instead of AWS Location Service | Free; one config change to swap later |
| Simulated telematics | Instead of provider integration | Avoids API contracts before product validation |
| Prisma | ORM | Migrations and schema work identically on Supabase and Aurora |
| Same frontend libs | MapLibre, Deck.gl, ECharts | Zero rework when upgrading to production stack |

## Migration Path to Production

When stakeholders approve, these are the steps — in order:

1. Split API routes into Fastify microservices (keep Prisma schemas)
2. Replace Supabase with Aurora PostgreSQL (connection string swap in Prisma)
3. Add Timestream for position history (keep PostgreSQL for all other tables)
4. Add ElastiCache Redis for live position cache
5. Replace Supabase Realtime with Kinesis → WebSocket Gateway
6. Replace OSM tiles with AWS Location Service (one line in MapLibre config)
7. Add Cognito with full SAML federation (replace Supabase Auth)
8. Add Kinesis + telematics adapter services
9. Containerise and deploy to EKS

Steps 1–4 can happen incrementally without rewriting the frontend.
Full details in CLAUDE.enterprise.md.

## Known Gotchas

- Supabase free tier pauses after 7 days of inactivity — run `pnpm db:seed`
  before any demo to wake it, or upgrade to Pro ($25/month) before the stakeholder session
- Supabase Realtime has a 200 concurrent connection limit on the free tier —
  fine for a demo, not for production
- MapLibre GL JS requires WebGL — test on the presentation machine beforehand
- Deck.gl `ScatterplotLayer` needs `coordinateSystem: COORDINATE_SYSTEM.LNGLAT`
  set explicitly or vehicle markers render at wrong positions
- Prisma requires `DIRECT_URL` for `prisma migrate` — the pooler (port 6543)
  does not support migrations
- Vercel hobby tier has a 10-second function timeout — keep API routes fast;
  run heavy seed queries locally, not via API routes

### Map rendering (hard-won lessons)

**MapLibre CSS must be imported globally.** Import `maplibre-gl/dist/maplibre-gl.css`
in `app/layout.tsx` — not inside the component. CSS inside a `next/dynamic` component
with `ssr: false` may be omitted from the bundle.

**Give the MapLibre container explicit block dimensions.** The container div must have
real `clientWidth` / `clientHeight` values when `new maplibregl.Map({ container: el })`
is called. `absolute inset-0` alone can resolve to `clientHeight = 0` in some
environments, causing MapLibre to fall back to the 300 px canvas default (the HTML
`<canvas>` default). Use `className="absolute inset-0 w-full h-full"` on the ref div.

**Call `map.resize()` inside the `load` handler.** The style sometimes loads before
the browser has finished painting the layout. Calling `map.resize()` inside
`map.on("load", ...)` forces MapLibre to re-read the container dimensions.

**`flex-1` containers need `min-w-0`.** If the map page is inside a flex layout
(e.g. sidebar + main), the `<main>` element needs `min-w-0` alongside `flex-1`,
otherwise flex children collapse to their minimum content width and the map
container gets a near-zero width.

### Supabase / Prisma connection strings

**Always use `DATABASE_URL` (pooler, port 6543) for application queries.**
Use `DIRECT_URL` (port 5432) only for `prisma migrate` / `prisma db push`.
The direct connection is unreachable from Vercel serverless functions and from
some local network configurations (IPv6 routing). Both `lib/prisma.ts` and
`scripts/simulate.ts` must use `DATABASE_URL`.

**The seed script must create Supabase Auth users.** `prisma.user.createMany()`
writes to the application `User` table, but login uses `supabase.auth.signInWithPassword`
which reads the Supabase Auth schema. Use `supabaseAdmin.auth.admin.createUser()` with
`email_confirm: true` for every demo account after inserting the DB rows.

### Vercel Hobby plan limits

**Cron jobs are daily-only.** `* * * * *` (per-minute) crons are blocked on the
Hobby plan. Remove cron config from `vercel.json` and run `pnpm simulate` locally
during demos. Upgrade to Pro if automated simulation on Vercel is required.

**API routes must not swallow errors.** Catch blocks that return a generic 500
without logging `err` make production debugging impossible. Always
`console.error("[route] error:", err)` and include `err.message` in the response
body during development.

## Out of Scope for MVP

Do not implement any of these — they belong in CLAUDE.enterprise.md:

- Any AWS services
- Microservices or separate API servers
- Real telematics provider integration
- Kinesis, MSK, Redis, Timestream, Redshift, OpenSearch
- gRPC or Protocol Buffers
- Docker or Kubernetes
- Multi-region data residency or GDPR deletion workflows
- PDF report generation
- Native mobile app
